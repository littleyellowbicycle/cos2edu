# 文件上传与 RAG 技术方案

## 一、方案对比

### 1.1 两种路线

| 路线 | 组件 | 适用场景 |
|------|------|----------|
| **自建文件上传 + RAG** | 自建上传 + LlamaIndex | 深度定制、完全控制 |
| **集成 RAGFlow** | RAGFlow 一站式 | 快速部署、功能完整 |

### 1.2 RAGFlow 优势

RAGFlow 是完整的 RAG 解决方案，包含：
- 文档上传与管理
- 文本解析与智能分割
- Embedding 生成
- 向量存储（内置）
- 检索与问答 UI

**如果采用 RAGFlow，不再需要自建文件上传系统。**

## 二、自建文件上传方案（备选）

> 仅在不使用 RAGFlow 时需要

### 2.1 当前问题
- 前端 `FileReader.readAsText()` 直接读取
- 内容存入数据库 `content` 字段
- **风险**：无防护、内存占用大、无病毒检测

### 2.2 增强方案

#### 2.2.1 文件存储
- 存储到服务器专用上传目录
- 随机 UUID 作为文件名，不用用户上传的名称
- 目录不可执行
- 不压缩（HTTP 传输时自动 gzip）

#### 2.2.2 病毒防护
- **MIME 类型校验**：前端 + 后端双重
- **Magic Number 校验**：检测文件真实头字节（如 PDF 是 `%PDF`）
- **白名单扩展名**：`.txt`, `.md`, `.pdf`, `.doc`, `.docx`
- **文件大小限制**：前端 50MB，后端校验
- **不执行文件**：仅存储和读取

### 2.3 风险说明
| 风险类型 | 评估 | 防护措施 |
|----------|------|----------|
| 传统病毒 | 低 | 白名单 + 不执行 |
| 提示词注入 | 中 | 输入过滤 + LLM 指令约束 |
| MIME 伪造 | 中 | Magic Number 校验 |
| Zip 炸弹 | 低 | 文件大小限制 |

## 三、RAG 技术方案

### 3.1 为什么需要 RAG

| 问题 | 传统方案 | RAG 方案 |
|------|----------|----------|
| Token 消耗 | 整篇发送，成本高 | 只发送相关片段 |
| 回答精准度 | 可能跑题 | 检索相关内容 |
| 可扩展性 | 难扩展新文档 | 随时添加 |

### 3.2 开源方案对比

| 方案 | 特点 | 适用场景 |
|------|------|----------|
| LangChain / LlamaIndex | 框架，灵活 | 需要深度定制 |
| **RAGFlow** | **一站式 RAG，功能完整** | **快速部署** |
| Quivr | 私人 LLM 助手 | 个人知识库 |
| MaxKB | 知识库问答，**中文友好** | 企业内部知识库 |

### 3.3 教育场景的特殊问题

| 问题 | 说明 | 解决方案 |
|------|------|----------|
| 上下文割裂 | 分割打断解释逻辑 | 语义分割，保留完整概念 |
| 教学结构丢失 | 教育有先验知识、递进关系 | 保留章节、难度元数据 |
| 连续推理断层 | 数学证明、编程教学需连贯 | 父子块关系，保持层次 |
| 跨章节引用 | 概念常引用其他章节 | 检索时按教学顺序重组 |
| 问答匹配度 | 学生提问与 chunk 不匹配 | 相关性检索 + 排序 |
| 碎片化理解 | 学生需整体理解 | 返回时提供上下文摘要 |

### 3.4 RAGFlow 教育适配

RAGFlow 支持：
- 自定义分割模板
- 父子块分割
- 元数据注入
- 可针对教育场景配置分割策略

## 四、决策建议

### 4.1 推荐路线

**采用 RAGFlow**：
- 零文件上传开发量
- 文档安全由 RAGFlow 处理
- 专注教育场景的检索策略定制

### 4.2 自建场景

如果需要完全自建（不用 RAGFlow）：
- 文件上传 + LlamaIndex + Chroma
- 白名单 + Magic Number 校验
- 自定义教育分割模板
