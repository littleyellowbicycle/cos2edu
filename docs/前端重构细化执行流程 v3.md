### 1.1 后端：LLMProvider 支持 Function Calling

**这是整个方案最关键的后端改动**。当前 `chat_stream()` 不传 `tools` 参数。

**修改文件**：`backend/app/services/chat_service.py`

```
LLMProvider.chat_stream() 签名变更：
  async def chat_stream(
      self,
      messages: List[Dict[str, str]],
      tools: Optional[List[Dict]] = None,    # 新增
  ) -> AsyncIterator[str | dict]:            # 返回类型变更：str 或 dict

新增工具调用的流式解析：
  OpenAI 格式：
    - stream=True 时 tool_calls 是增量式，需累积拼接
    - 每个 chunk.delta.tool_calls[i].function.name / arguments（部分）
    - 完整 tool_call 拼接后，yield {"type": "tool_calls", "tool_calls": [...]}

  Anthropic 格式：
    - tool_use 是 content block（非增量）
    - 在 stream 完成后处理

  本地模型（Ollama 等）：
    - 可能不支持 tool_calls
    - 检测 models/ 容量，不支持时降级为纯文本模式
```

**关键实现细节**：

```python
# OpenAI 流式 tool_calls 增量拼接逻辑（伪代码）
tool_calls_accumulator = {}  # {index: {name: str, arguments: str}}
async for chunk in stream:
    delta = chunk.choices[0].delta
    
    # 文本内容照常 yield
    if delta.content:
        yield {"type": "text", "content": delta.content}
    
    # tool_calls 增量拼接
    if delta.tool_calls:
        for tc in delta.tool_calls:
            idx = tc.index
            if idx not in tool_calls_accumulator:
                tool_calls_accumulator[idx] = {"name": "", "arguments": ""}
            if tc.function.name:
                tool_calls_accumulator[idx]["name"] += tc.function.name
            if tc.function.arguments:
                tool_calls_accumulator[idx]["arguments"] += tc.function.arguments
    
    # 流结束时，如果累积了 tool_calls，一次性 yield
    if chunk.choices[0].finish_reason == "tool_calls":
        yield {
            "type": "tool_calls",
            "tool_calls": [
                {"name": tc["name"], "arguments": tc["arguments"]}
                for tc in tool_calls_accumulator.values()
            ]
        }
```

**验证**：用 GPT-4o 或兼容模型发送带 `tools` 参数的请求，确认流式返回中正确解析文本和 tool_calls。

### 1.9 Narrative Store 扩展

**修改文件**：`frontend/src/stores/narrative.js`

新增状态：
```javascript
generativeComponents: ref([]),  // 当前活跃的 Generative 组件列表

// Actions
addGenerativeComponent(comp) {
  // 去重：同一 comp_id 不重复添加
  const idx = this.generativeComponents.findIndex(c => c.id === comp.id)
  if (idx >= 0) this.generativeComponents[idx] = comp
  else this.generativeComponents.push(comp)
},
removeGenerativeComponent(compId) {
  this.generativeComponents = this.generativeComponents.filter(c => c.id !== compId)
},
updateGenerativeComponent(compId, newProps) {
  const comp = this.generativeComponents.find(c => c.id === compId)
  if (comp) Object.assign(comp.props, newProps)
},
```

## 关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM 不支持 Function Calling | 无法触发 Generative UI | 后端检测模型能力：不支持 tools 时降级为纯文本模式；保留现有 WS 事件路径作为 fallback |
| 流式 tool_calls 解析复杂 | OpenAI 流式响应中 tool_calls 增量拼接容易出错 | 先实现非流式 tool_calls（stream=False）验证端到端，再改为流式 |
| Anthropic tool_use 格式不同 | 需要单独适配 | LLMProvider 按 provider 分支处理 |
| Chat.vue 拆分引入回归 | 1827 行单体的拆分是高风险操作 | 每拆一个子组件就做端到端验证；保留 fallback |
| 两套 CSS 变量冲突 | 页面视觉回归 | 阶段 0 先统一，再动组件 |
| TailwindCSS 与 Element Plus 冲突 | 样式优先级覆盖 | 关闭 preflight；新组件用 Tailwind，旧组件保留手写 CSS |
| 前端无测试 | 重构无法验证功能无回归 | 阶段 0 不强求 E2E 测试，但每个子组件拆分后做手动端到端验证 |

## 重构期间功能稳定性保障策略

### 核心原则：双路径并存 + 增量切换

不是一次性替换，而是新增路径与旧路径并行运行，新路径稳定后再移除旧代码。

### 1. Generative UI 管线 — 新旧并行

现有 WS 事件路径（`emotion.update`、`assessment.quiz`、`scene.change` 等）**保留不动**。Generative UI 是额外新增一条路径：

```
旧路径（保留）：后端 → emotion.update WS → Chat.vue 直接渲染
新路径（新增）：后端 → ui.render WS → DynamicRenderer → EmotionCard

Chat.vue 模板加条件判断：
  有 generative 组件 → 用 DynamicRenderer 渲染
  没有 → 走原有逻辑
```

只有当新路径稳定后，才逐步移除旧路径。

### 2. LLM Function Calling — 降级兜底

```python
# narrative_engine.py 流式输出逻辑
if tools_supported and model_config.supports_tools:
    # 新路径：送 tools 参数，解析 tool_calls
    stream = llm.chat_stream(messages, tools=ui_orchestrator.TOOL_DEFINITIONS)
else:
    # 旧路径：纯文本，跟现在完全一样
    stream = llm.chat_stream(messages)
```

不支持 tools 的模型（如某些本地模型）自动降级，零影响。

### 3. Chat.vue 拆分 — 逐个提取 + 验证

每次只提取一个子组件，提取后立即验证：

```
第1步：提取 MarkdownContent.vue → Chat.vue 引入子组件 → 验证消息渲染正常
第2步：提取 MessageBubble.vue → 验证正常
第3步：提取 ChatInput.vue → 验证正常
第4步：提取 ChatHeader.vue → 验证正常
第5步：提取 ChatSidebar.vue → 验证正常
第6步：ChatMessages.vue（消息列表滚动逻辑）→ 验证正常
```

### 4. CSS 变量统一 — 先合并再切换

```css
/* variables.css — 同时保留两套变量名 */
:root {
  /* 新命名（Chat.vue 使用的） */
  --color-bg: #FAF8F5;
  --color-surface: #FFFFFF;

  /* 旧命名兼容（其他页面使用的） */
  --bg-deep: #0d0c0a;
  --bg-surface: #151412;
  --amber-glow: #d4a574;
}
```

先合并到一个文件 → 逐页面迁移变量名 → 全部迁移后删除旧名。过程中两套名始终可用。

### 6. 向后兼容策略汇总

| 改动 | 旧路径 | 新路径 | 兼容方式 |
|------|--------|--------|---------|
| Generative UI 渲染 | Chat.vue 直接渲染 assessment/event/emotion | DynamicRenderer 接收 ui.render | 条件判断：有 generative 组件走新路径，否则走旧路径 |
