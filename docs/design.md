1. 教材存储：当前设计的缺失
回顾刚才生成的架构文档，确实存在一个明显的遗漏——教材/课程内容没有明确的存储位置。当前设计中：

teaching_log.md 只记录了"教了什么、效果如何"，是教学过程的日志
角色配置中的 teaching_style 只定义了"怎么教"，是教学风格
但教什么——课程大纲、知识点体系、练习题库、参考资源——完全没有着落
这意味着三位老师每次上课，LLM 需要凭空生成教学内容，既无法保证知识覆盖的完整性，也无法确保难度梯度的合理性。

应该补充的设计：

cos2edu/
+-- curriculum/                  # 课程体系目录（新增）
|   +-- syllabus.yaml            # 课程大纲：3个月学习计划
|   +-- modules/                 # 知识模块
|   |   +-- 01_machine_learning_basics.yaml
|   |   +-- 02_neural_networks.yaml
|   |   +-- 03_deep_learning.yaml
|   |   +-- 04_nlp.yaml
|   |   +-- ...
|   +-- exercises/               # 练习题库
|   |   +-- week_01.yaml
|   |   +-- week_02.yaml
|   |   +-- ...
|   +-- resources/               # 参考资源
|   |   +-- reading_list.yaml    # 推荐阅读
|   |   +-- datasets.yaml        # 练习数据集
每个知识模块的 YAML 大致长这样：


# curriculum/modules/02_neural_networks.yaml
id: neural_networks
name: 神经网络基础
order: 2
estimated_days: 8
prerequisites: [machine_learning_basics]

knowledge_points:
  - id: perceptron
    name: 感知机
    difficulty: 2
    key_concepts: [线性分类, 激活函数, 损失函数]
    teaching_hints:
      ganyu_style: 从MP神经元的历史讲起，建立直觉
      keqing_style: 直接给公式，然后推导收敛性
      march7th_style: 先写代码实现一个，再解释原理

  - id: backpropagation
    name: 反向传播算法
    difficulty: 4
    key_concepts: [计算图, 链式法则, 梯度消失]
    teaching_hints:
      ganyu_style: 从数学角度严格推导
      keqing_style: 总结计算步骤，给解题模板
      march7th_style: 用PyTorch自动求导演示

exercises:
  - type: calculation
    description: 手动计算三层网络的前向传播和反向传播
    difficulty: 3
  - type: coding
    description: 用NumPy实现一个简单的MLP
    difficulty: 4
关键设计点：teaching_hints 字段为不同教学风格的角色提供了针对同一知识点的差异化教学建议，这样教学协作引擎在做动态分工时，就有了比 teaching_style 标签更精细的依据。

2. 前后端分离：当前设计的另一个盲区
当前架构文档只描述了后端引擎和数据层，完全没有涉及前端。但一个叙事驱动的教育系统，用户需要：

需求	为什么纯后端不够
对话界面	用户需要和角色聊天
时间线可视化	用户需要看到90天倒计时的进度
角色状态感知	用户需要感知角色的情感变化（不一定直接看Markdown）
场景切换	用户需要选择或被引导到不同场景
事件通知	时间/条件/随机事件需要以叙事方式呈现给用户
考核界面	考试和辩论赛需要专门的交互界面
所以答案是：必须前后端分离。 但不是传统的 CRUD 分离，而是一种更适合叙事系统的架构：

几个关键设计决策：

通信方式：WebSocket 为主，REST 为辅。 对话是流式的（LLM 逐字输出），事件通知是推送式的（不能等用户来轮询），所以 WebSocket 是必须的。REST 只用于非实时的操作，如查看历史记录、修改角色状态等。

前端叙事状态管理： 前端维护一个精简的世界状态镜像（当前时间、场景、角色心情摘要、关系概况），用于驱动 UI 渲染。完整的状态文档只存在于后端，前端只获取渲染所需的摘要信息。这样既保证了前端的响应速度，又避免了将完整的角色内心世界暴露给用户（保留叙事的神秘感）。

教材的归属： 课程体系（curriculum/）是纯后端数据，前端不直接访问。教学协作引擎根据课程大纲和用户的学习进度，决定每次教学对话的主题和难度，然后通过 Prompt 注入给 LLM。前端只需要展示"当前学习进度"和"下次课程预告"等摘要信息。
