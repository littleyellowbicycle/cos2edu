## 阶段 0：工程基础设施准备（前置条件）

> **目标**：在动手重构之前，建立必要的工程化基础设施，确保后续每个阶段都能安全迭代。

### 0.1 引入 TypeScript（渐进式）

**为什么**：v2 方案中的 `registry.ts`、`GenerativeComponentMeta` 等核心接口依赖类型系统；当前项目全部是 `.js`，无法获得类型安全。

**执行步骤**：

1. **安装依赖**
   ```bash
   cd frontend
   npm install -D typescript @types/node vue-tsc
   ```

2. **创建 `tsconfig.json`**
   ```json
   {
     "compilerOptions": {
       "target": "ES2020",
       "module": "ESNext",
       "moduleResolution": "bundler",
       "strict": false,
       "jsx": "preserve",
       "allowJs": true,
       "noEmit": true,
       "paths": { "@/*": ["./src/*"] },
       "types": ["vite/client"]
     },
     "include": ["src/**/*.ts", "src/**/*.d.ts", "src/**/*.vue"]
   }
   ```

3. **创建 `src/env.d.ts`**（Vue SFC 类型声明）
   ```typescript
   /// <reference types="vite/client" />
   declare module '*.vue' {
     import type { DefineComponent } from 'vue'
     const component: DefineComponent<{}, {}, any>
     export default component
   }
   ```

4. **Vite 配置添加 Vue TSC 插件**（`vite.config.js`）

5. **策略：增量迁移**——新文件全部用 `.ts`，旧文件按需迁移（不强求一次性全部迁移）

**验证标准**：`npx vue-tsc --noEmit` 无报错；现有 `.js` 文件正常运行。

### 0.2 统一 CSS 变量体系

**为什么**：当前 `App.vue` 和 `global.css` 存在两套冲突的 CSS 变量（浅色 `--color-bg` vs 深色 `--bg-deep`），后续引入 TailwindCSS 和新组件前必须先统一。

**执行步骤**：

1. **确定主主题**：根据 Chat.vue 实际使用的变量（`--color-bg`、`--color-surface` 等），确定以 App.vue 的浅色变量为基础
2. **合并到 `src/styles/variables.css`**：统一变量命名，删除重复定义
3. **更新 `global.css`**：移除冲突变量，Element Plus 覆盖样式保留但改用统一变量
4. **逐页面检查**：确保所有视图组件的 CSS 引用统一后的变量名

**验证标准**：所有页面视觉无回归；CSS 变量命名一致。

### 0.3 引入 ESLint + Prettier

**执行步骤**：

1. 安装 `eslint` + `@vue/eslint-config-typescript` + `prettier`
2. 创建 `.eslintrc.cjs` 和 `.prettierrc`
3. 在 `package.json` 添加 `lint` 和 `format` 脚本

**验证标准**：`npm run lint` 通过。

## 阶段 1：Generative UI 基础设施（核心管线搭建）

> **目标**：建立从后端 LLM Function Calling → UI JSON → 前端动态渲染的完整管线，用 EmotionCard 做端到端验证。

### 1.1 前端：组件注册表（registry.ts）

**新建文件**：`frontend/src/components/generative/registry.ts`

**实现内容**：

```typescript
interface PropDefinition {
  type: 'string' | 'number' | 'boolean' | 'array' | 'object'
  required: boolean
  enum?: string[]
  min?: number
  max?: number
  default?: any
}

interface GenerativeComponentMeta {
  name: string
  version: string
  description: string
  propsSchema: Record<string, PropDefinition>
  slot: 'inline' | 'sidebar' | 'overlay' | 'panel'
  defaultLifecycle: 'persistent' | 'ephemeral' | 'sticky'
  loader: () => Promise<any>
}

// 核心函数：registerComponent / getComponent / validateProps / getAllComponents
```

**关键决策**：
- 使用 `Map<string, GenerativeComponentMeta>` 存储，O(1) 查找
- `validateProps` 根据 `propsSchema` 校验后端推送的 props，防止渲染错误
- `loader` 使用动态 `import()` 实现懒加载

### 1.2 前端：DynamicRenderer 组件

**新建文件**：`frontend/src/components/generative/DynamicRenderer.vue`

**实现内容**：
- Props: `slot`（渲染位置标识）
- 监听 WebSocket `ui.render` 事件
- 根据 `component` 名从 registry 查找 → `defineAsyncComponent` 异步加载 → 渲染
- 生命周期管理：`persistent` 组件持久存在，`ephemeral` 随消息消失，`sticky` 置顶直到替换
- 错误边界：加载失败时显示 ErrorFallback 组件
- Loading 状态：加载中显示 LoadingPlaceholder

**关键决策**：
- 每个 slot 位置一个 DynamicRenderer 实例（inline / sidebar / overlay / panel）
- 组件 ID 去重：同一 `comp_id` 不重复渲染
- 组件交互回调：`@interact` 事件通过 WebSocket 发送 `ui.interact` 消息回后端

### 1.3 前端：扩展 WebSocket 协议

**修改文件**：`frontend/src/composables/useWebSocket.js`

**新增消息类型**：

| 类型 | 方向 | 说明 |
|------|------|------|
| `ui.render` | S→C | 推送 Generative 组件渲染指令 |
| `ui.destroy` | S→C | 销毁指定组件 |
| `ui.update` | S→C | 更新指定组件的 props |
| `ui.interact` | C→S | 用户与组件交互（如提交测验） |

**新增方法**：
- `sendUIInteract(componentId, action, value)` — 发送组件交互事件

**修改点**：
- `_dispatch` 方法已支持任意 `msg.type`，无需修改
- 只需在 Chat.vue 的 `onMounted` 中新增 `ws.on('ui.render', ...)` 等监听

### 1.4 后端：UI Orchestrator

**新建文件**：`backend/app/engines/ui_orchestrator.py`

**实现内容**：

1. **`TOOL_DEFINITIONS`**：5 个 Function Calling 工具定义
   - `show_emotion_card` → EmotionCard
   - `show_timeline` → Timeline
   - `show_quiz` → QuizForm
   - `show_knowledge_graph` → KnowledgeGraph
   - `switch_scene` → SceneCard

2. **`TOOL_TO_COMPONENT`**：工具名 → (组件名, slot, lifecycle) 映射

3. **`convert_tool_calls(tool_calls)`**：将 LLM 返回的 tool_calls 转换为 `UIComponent` 列表

4. **`build_ui_render_message(components)`**：组装 `ui.render` WebSocket 消息

**Pydantic 模型**：
```python
class UIComponent(BaseModel):
    id: str
    component: str
    version: str = "1.0"
    props: dict
    slot: ComponentSlot
    position: str = "after_text"
    lifecycle: ComponentLifecycle

class UIRenderMessage(BaseModel):
    type: str = "ui.render"
    id: str
    timestamp: str
    components: list[UIComponent]
    metadata: dict = {}
```

### 1.5 后端：修改 LLM 调用链路（集成 Function Calling）

**修改文件**：`backend/app/services/chat_service.py` + `backend/app/engines/narrative_engine.py`

**这是最关键的后端改动**。当前 `ChatService.chat_stream()` 只传 `messages` 给 LLM，不传 `tools`。需要：

1. **`LLMProvider.chat_stream()` 新增 `tools` 参数**：
   ```python
   async def chat_stream(
       self,
       messages: List[Dict[str, str]],
       tools: Optional[List[Dict]] = None,  # 新增
   ) -> AsyncIterator[str | dict]:
   ```

2. **OpenAI 兼容接口**：在 `client.chat.completions.create()` 中传入 `tools=tools` 和 `stream=True`

3. **流式解析 tool_calls**：OpenAI 流式响应中 tool_calls 是增量式的，需要累积拼接：
   ```python
   # 流式 tool_call 增量拼接逻辑
   tool_call_accumulator = {}
   async for chunk in stream:
       if chunk.choices[0].delta.content:
           yield {"type": "text", "content": chunk.choices[0].delta.content}
       if chunk.choices[0].delta.tool_calls:
           for tc_chunk in chunk.choices[0].delta.tool_calls:
               # 累积 tool_call 的 name 和 arguments
               ...
   ```

4. **`NarrativeEngine.handle_chat_message()` 集成 UIOrchestrator**：
   ```python
   # 在调用 LLM 之前
   tools = self.ui_orchestrator.TOOL_DEFINITIONS
   
   # 在流式输出过程中
   async for event in llm.chat_stream(messages, tools=tools):
       if event["type"] == "text":
           yield json.dumps({"type": "chat.token", "content": event["content"]})
       elif event["type"] == "tool_calls":
           ui_components = self.ui_orchestrator.convert_tool_calls(event["tool_calls"])
           if ui_components:
               yield json.dumps({
                   "type": "ui.render",
                   "components": [c.model_dump() for c in ui_components],
                   ...
               })
   ```

5. **Anthropic 兼容**：Anthropic 的 tool_use 格式不同，需要单独处理（`tool_use` content block）

### 1.6 后端：扩展 WebSocket 协议 Schema

## 阶段 2：核心 Generative 组件开发

> **目标**：实现 5 个核心 Generative 组件，逐步替换 Chat.vue 中的硬编码 UI。

### 2.1 QuizForm（交互式测验）

**新建文件**：`frontend/src/components/generative/QuizForm.vue`

**功能**：
- 接收 props: `{ pointId, questions, questionType, options? }`
- 支持三种题型：选择题（radio）、判断题（true/false）、简答题（textarea）
- 提交按钮 → 触发 `@interact` 事件 → WebSocket `ui.interact` → 后端处理
- 显示测验结果（由后端推送 `ui.update` 更新 props 显示结果）

**迁移策略**：
- 替换 Chat.vue 中硬编码的 `assessment-panel`（第 70-91 行）和 `assessment-result`（第 93-104 行）
- 后端 `assessment.generate` 和 `assessment.answer` 消息类型保留，但改为通过 UIOrchestrator 推送
- 新增 `show_quiz` Function Calling 工具，让 LLM 可以主动触发测验

**注意**：QuizForm 是双向交互组件，需要处理 `ui.interact` 回传和 `ui.update` 更新结果。

### 2.2 Timeline（历史事件时间线）

**新建文件**：`frontend/src/components/generative/Timeline.vue`

**功能**：
- 接收 props: `{ events: [{year, label}], title? }`
- 水平/垂直时间线布局
- 事件节点可点击，点击后发送 `ui.interact` 请求详情
- slot: `sidebar`，lifecycle: `sticky`

**依赖**：无需额外图表库，纯 CSS + Vue 实现。

### 2.3 KnowledgeGraph（知识点依赖关系图）

**新建文件**：`frontend/src/components/generative/KnowledgeGraph.vue`

**功能**：
- 接收 props: `{ nodes, edges, highlight? }`
- DAG 可视化：节点 + 有向边
- 高亮当前知识点，已掌握的节点变灰
- 支持缩放和拖拽
- slot: `panel`，lifecycle: `sticky`

**依赖**：需要引入轻量级图可视化库。

**技术选型**：
- 方案 A：**D3.js**（灵活但学习曲线高）
- 方案 B：**@vue-flow/core**（Vue 3 专用流程图库，推荐）
- 方案 C：纯 SVG 手写（简单 DAG 够用）

**推荐方案 B**：`@vue-flow/core` 是 Vue 3 原生的节点图库，API 简洁，支持拖拽缩放。

```bash
npm install @vue-flow/core @vue-flow/background @vue-flow/controls
```

### 2.4 SceneCard（场景切换卡片）

**新建文件**：`frontend/src/components/generative/SceneCard.vue`

**功能**：
- 接收 props: `{ sceneId, sceneName, description, backgroundImage?, transition? }`

- slot: `overlay`，lifecycle: `ephemeral`（2-3 秒后自动消失）

**迁移策略**：

### 2.5 EmotionCard 增强

**核心问题**：Tauri 壳启动时需要同时启动 FastAPI 后端。

3. **启动逻辑**：在 Tauri Rust 代码中，应用启动时 spawn sidecar 进程，应用退出时 kill

## 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM 不支持 Function Calling（如某些本地模型） | 无法触发 Generative UI | 后端检测模型能力，不支持时走纯文本模式；Ollama 模型需确认 tool_support |
| 流式 tool_calls 解析复杂 | OpenAI 流式响应中 tool_calls 是增量的，拼接逻辑容易出错 | 先实现非流式 tool_calls（`stream=False`），验证端到端后再改为流式 |
| Anthropic tool_use 格式不同 | 需要单独适配 | 在 `LLMProvider` 中按 provider 分支处理 |
| Chat.vue 拆分可能引入回归 | 消息渲染、WS 事件监听等逻辑分散后可能遗漏 | 每拆一个子组件就做端到端验证 |
| TailwindCSS 与 Element Plus 样式冲突 | 优先级覆盖问题 | TailwindCSS 的 `preflight` 关闭，或用 `important` 策略 |
| Tauri Sidecar 打包体积大 | PyInstaller 产物可能 100MB+ | 考虑远程后端模式作为备选 |
